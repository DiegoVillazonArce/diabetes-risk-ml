## D-013 Artifact Distribution Comparison (P7, 2026-07-11)

Formal comparison behind the D-013 acceptance, recorded before the first public deployment (US-0702). The evaluated artifact is the P6 single-bundle `models/diabetes_risk_model.joblib` (~263 KB, fitted D-016 model plus serving metadata). Sources consulted on 2026-07-11: Streamlit Community Cloud documentation ([app dependencies](https://docs.streamlit.io/deploy/streamlit-community-cloud/deploy-your-app/app-dependencies), [file organization](https://docs.streamlit.io/deploy/streamlit-community-cloud/deploy-your-app/file-organization), [deploy settings](https://docs.streamlit.io/deploy/streamlit-community-cloud/deploy-your-app/deploy)), [scikit-learn model persistence](https://scikit-learn.org/stable/model_persistence.html), and [GitHub large-file limits](https://docs.github.com/en/repositories/working-with-files/managing-large-files/about-large-files-on-github).

Platform facts that frame the comparison: Community Cloud copies the repository files and runs `streamlit run` from the repository root on Debian Linux; dependencies install from `requirements.txt`; there is no pre-runtime build step, hook, or arbitrary script execution; Python 3.12 is the platform default version. scikit-learn supports loading persisted models only from trusted, verified sources into an environment with the same library versions. GitHub warns at 50 MiB per file and blocks 100 MiB.

| Criterion | A. Controlled Git exception (selected) | B. GitHub Release asset | C. Generation during deployment build |
|---|---|---|---|
| Artifact size | ~263 KB: ~0.5% of GitHub's 50 MiB warning threshold; negligible clone cost. | Fine (releases target large files), but the size benefit is moot at 263 KB. | No stored artifact, but the 22.7 MB git-ignored CSV would be needed instead. |
| Reproducibility | Official artifact is pinned in history next to the exact code and pins that produced it; regenerable via `python -m src.artifacts` (fixed seed, pinned versions). | Asset must be built, uploaded, and kept in sync with the repo manually; drift between repo pins and released artifact is possible. | Would retrain per deploy; deterministic in theory (fixed seed) but unverifiable on this platform because no build step exists. |
| Security / provenance of the joblib | Provenance is the repository itself (trusted source per scikit-learn guidance); load-time validator checks model class, hyperparameters, feature order, and classes. | Requires download plus checksum/provenance verification logic; the project's durable local-only loading policy forbids remote artifact sources in the app. | No download, but training code would run inside the serving process with secret Kaggle credentials present. |
| Version compatibility | Artifact and `requirements.txt` travel in the same commit; metadata records Python/numpy/pandas/scikit-learn/joblib versions for verification. | Released artifact can lag behind repo pin changes; compatibility must be re-checked out of band. | Always trained with installed versions -- the one real advantage, unusable here. |
| Git history impact | ~263 KB per regeneration; acceptable while regenerations stay tied to contract/dependency changes. | None (clean history). | None. |
| Maintenance | Lowest: one file, one workflow, no extra infrastructure. | Release workflow, tagging, checksums, and sync discipline per model update. | Kaggle credential management, download scripting, and platform workarounds. |
| Network dependency | None beyond cloning the repo (which deployment does anyway). | Runtime download from GitHub on cold start; outage or rate limit breaks startup. | Runtime download of the CSV from Kaggle (authenticated) on cold start. |
| Cold start | Fastest: file already present when the app starts. | Slower: fetch + verify before first prediction. | Slowest: download 22.7 MB CSV + train (~minutes) before first prediction. |
| CSV availability | Not needed at deploy time; training stays local/offline. | Not needed at deploy time. | Blocker: the CSV is git-ignored (D-006) and absent from the deployment copy; obtaining it needs Kaggle credentials as platform secrets. |
| Streamlit Community Cloud viability | Native fit: the platform copies repository files, so the committed artifact simply arrives. | Only via in-app runtime download, since no pre-runtime hook exists. | Not viable: no build step exists, so "build-time" generation would actually run inside the Streamlit serving process. |
| Build/training vs. runtime serving separation | Preserved: training remains an offline local step (D-007); the app only loads a file. | Preserved for training, but adds a runtime distribution step inside the app. | Broken: training would execute in the serving process at startup. |

Alternative C is rejected because the platform offers no reproducible, secure pre-runtime mechanism (the mandate's own rejection criterion), it would require the git-ignored CSV or credentials at deploy time, and it would break the offline-training/serving separation. Alternative B is rejected because, without a build step, the download must happen inside the app at runtime -- adding a cold-start network dependency and checksum/provenance machinery that the project's local-only artifact-loading policy explicitly forbids, to solve a distribution problem that a 263 KB file does not have. Alternative A is selected: it is the only alternative that needs no new mechanism, keeps provenance in the repository, keeps the app local-only, and matches how the platform actually delivers files.
